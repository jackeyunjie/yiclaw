# OpenClaw 个人 AI 助手 - 需求对齐文档

## 1. 项目概述

### 1.1 项目目标
构建一个类似 OpenClaw 的个人 AI 助手系统，具备以下核心能力：
- 多通道消息接入（微信、Telegram、WebChat 等）
- AI 智能对话与任务处理
- 语音交互能力
- 工具调用与扩展
- 本地优先、私有化部署

### 1.2 参考项目分析

#### OpenClaw 核心架构
```
┌─────────────────────────────────────────────────────────────┐
│                     Gateway (控制平面)                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│  │ Sessions │  │ Channels │  │  Tools   │  │  Events  │     │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘     │
└─────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│   WhatsApp    │    │   Telegram    │    │    Slack      │
└───────────────┘    └───────────────┘    └───────────────┘
```

#### OpenClaw 功能特性
| 模块 | 功能描述 |
|------|----------|
| **Gateway** | WebSocket 控制平面，管理会话、配置、定时任务 |
| **Channels** | 多通道接入：WhatsApp、Telegram、Slack、Discord、Signal、iMessage 等 |
| **Agent** | AI 运行时，支持工具流、块流式响应 |
| **Tools** | 浏览器、画布、定时任务、会话管理等 |
| **Voice** | 语音唤醒与对话（macOS/iOS/Android）|
| **Canvas** | 可视化工作空间 |
| **Security** | DM 配对机制、访问控制 |

## 2. 需求理解

### 2.1 核心功能需求

#### 2.1.1 必须实现 (MVP)
1. **Gateway 网关服务**
   - WebSocket 实时通信
   - 会话管理（用户会话隔离）
   - 配置管理
   - 日志与监控

2. **多通道接入**
   - WebChat（网页聊天界面）
   - 微信（个人号/企业微信）
   - Telegram Bot
   - 预留其他通道扩展接口

3. **AI 对话引擎**
   - 支持多模型（OpenAI、Claude、本地模型）
   - 流式响应
   - 上下文记忆
   - 系统提示词管理

4. **工具系统**
   - 工具注册与发现
   - 工具调用执行
   - 基础工具：时间、计算器、天气等

5. **Web 管理界面**
   - 配置管理
   - 会话查看
   - 日志监控
   - 通道管理

#### 2.1.2 进阶功能 (Phase 2)
1. **语音交互**
   - 语音识别（ASR）
   - 语音合成（TTS）
   - 语音唤醒

2. **高级工具**
   - 浏览器自动化
   - 文件系统操作
   - 代码执行环境

3. **Canvas 可视化**
   - 交互式画布
   - 图表渲染

4. **移动端支持**
   - iOS/Android App
   - 推送通知

### 2.2 技术约束

#### 2.2.1 运行环境
- **服务器**: 阿里云 ECS (2核2G, Windows Server 2022)
- **数据库**: 阿里云 RDS MySQL 8.0 (2核4G, 100GB)
- **开发环境**: Windows 本地开发
- **运行时**: Node.js ≥ 22

#### 2.2.2 技术栈选择
| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 后端 | Node.js + TypeScript | 与 OpenClaw 保持一致 |
| 框架 | Express/Fastify | API 服务框架 |
| 数据库 | MySQL 8.0 | 阿里云 RDS |
| ORM | Prisma/TypeORM | 数据库访问 |
| 实时通信 | Socket.io | WebSocket 支持 |
| 前端 | React + TypeScript | Web 管理界面 |
| AI SDK | OpenAI SDK / Anthropic SDK | 模型调用 |
| 部署 | PM2 | 进程管理 |

## 4. 疑问澄清

### 4.1 需要确认的问题

#### Q1: AI 模型选择
**问题**: 您希望使用哪些 AI 模型？
- OpenAI (GPT-4/GPT-3.5)
- Claude (Anthropic)
- 国产模型（通义千问、文心一言等）
- 本地部署模型（Ollama）

**建议**: 初期支持 OpenAI 和 Claude，预留扩展接口

#### Q2: 微信接入方式
**问题**: 微信通道您希望如何接入？
- 个人微信号（itchat/wechaty，可能有封号风险）
- 企业微信（官方 API，更稳定）
- 微信公众号

**建议**: 同时支持企业微信（稳定）和个人微信（体验）

#### Q3: 安全策略
**问题**: 访问安全策略如何设计？
- 是否需要用户认证系统？
- DM 消息是否需要配对机制？
- API 密钥如何管理？

**建议**: 
- Web 管理界面需要登录
- 通道消息默认开放（单用户场景）
- API 密钥使用 .env 文件管理

#### Q4: 数据存储
**问题**: 聊天记录和配置数据的存储策略？
- 聊天记录保留时长？
- 是否需要消息加密？
- 备份策略？

**建议**:
- 聊天记录默认保留 30 天
- 敏感配置加密存储
- 定期自动备份
